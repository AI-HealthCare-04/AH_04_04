## 1. 왜 이 데이터가 필요한가 (§5-5)

정상 보행 직후 곧바로 앉으면, 앉기 피크 간격이 ≈882 ms(≈68 보/분)로 **정상 보행 대역(500~1000 ms)
한가운데**라, 현재 감지기가 쓰는 **간격이라는 단일 신호로는 분리가 불가능**하다(20 보 + 앉기 5회 = 28 로
과다카운트). 현 감지기는 `sqrt(x²+y²+z²)` 스칼라만 쓰고 x/y/z 를 버리므로, 간격 외의 신호(축 방향·진폭·분산
등)를 보려면 **버려지던 원시 3축이 라벨과 함께** 있어야 한다. 합성 파형(순수 Z축)으로는 두 동작이 구분되지
않아 **실기기 실측이 유일한 게이트**다.

## 2. 수집 매트릭스 (라벨 4 × 배치 3 × 10회 ≈ 120 trial)

### 2-1. 라벨 (`WaveformLabel`)

| 라벨 | id | 앉기 큐 | 동작 | 역할 |
|---|---|---|---|---|
| 정상 보행 | `normal_walk` | ✗ | 총 20초 계속 걷기 | 대조군 — 오탐이 **나면 안 되는** 기준 |
| 보행 직후 앉기 | `walk_then_sit` | ✓ | 15초 걷기 → 비프 → 앉기 5초 | **오탐 대상**(§5-5) |
| 서서 앉기만 | `sit_only` | ✓ | 15초 서있기 → 비프 → 앉기 5초 | 보행 momentum 없는 **순수 앉기 하강**의 기준선 |
| 제자리 발끌기 | `shuffle` | ✗ | 총 20초 제자리 발끌기 | 대조군 — 작은 진폭의 애매한 움직임 |

- `sit_only` 는 앉기 신호 자체의 축·진폭 프로파일을, `walk_then_sit` 과 겹쳐 봄으로써 "보행이 섞였을 때
  앉기 신호가 어떻게 달라지는가"를 분리한다.
- `shuffle` 은 "보행은 아니지만 작게 흔들리는" 움직임이 정상 보행 판별에 끼치는 영향을 본다.

### 2-2. 배치 (`PlacementSpec` 프리셋 3종)

센서 좌표계는 기기 자연방향 기준이라 같은 위치라도 화면 방향·상단 방향에 따라 x/y/z 가 달라진다.
아래 3 프리셋으로 고정한다(수집 화면에서 선택 시 배치 안내가 그대로 표시됨).

| 프리셋 | 위치 | 좌/우 | 화면 | 상단 | 접힘 |
|---|---|---|---|---|---|
| 앞주머니(안) | front_pocket | right | in(몸쪽) | up | folded |
| 앞주머니(밖) | front_pocket | right | out(바깥) | up | folded |
| 손(세로) | hand | na | user | up | unfolded |

### 2-3. 횟수

- **조건당 10회** → 4 × 3 × 10 = **120 trial**.
- 각 trial 은 도구가 고정 타이밍으로 자동 진행·자동 종료하므로, 사람이 화면을 건드릴 필요가 없다.

## 3. 수집 절차 (기기 무접촉 고정 프로토콜)

> 📋 **현장에서 폰을 들고 그대로 따라 하는 실전 체크리스트는**
> [`sensor_waveform_capture_field_guide.md`](sensor_waveform_capture_field_guide.md)**에 있다.** 아래는 요약.

1. **미디어 음량 확보** → 화면의 `🔊 비프 테스트`로 비프가 실제로 들리는지 확인(주머니 배치 전).
2. 라벨·배치 프리셋 선택 → 안내대로 휴대폰을 둔다.
3. `● 녹화 시작` 후 **화면을 건드리지 않는다.**
   - 앉기 큐 라벨(`walk_then_sit`·`sit_only`): 15초 뒤 **비프**가 울리면 그때 앉는다 → 5초 뒤 자동 정지.
   - 큐 없는 라벨(`normal_walk`·`shuffle`): 20초 동작 후 자동 정지.
4. `저장 & 공유 (CSV)` 로 파일을 내보낸다(파일명 = `<label>_<yyyyMMdd_HHmmss_SSS>.csv`).

> **비프 전달 검증:** 앉기 큐 라벨은 큐 순간 "미디어 음량 > 0 && startTone 성공"을 확인한 경우에만
> `sit_cue` 마커를 찍고 `cue_delivery=success` 로 남긴다. 실패 시 녹화를 자동 중단한다.

### 고령/느린 보행 반영 (타깃 = 독립 보행 시니어)

- §5-5 문제 자체가 **느린 케이던스(≈68 보/분)**에서 발생하므로, 보행이 포함된 라벨
  (`normal_walk`·`walk_then_sit`)의 걷기는 **실제 고령자 보폭·속도(느린 편)**로 수행한다.
- 빠른 케이던스가 필요하면 후속 수집으로 별도 라운드를 추가한다(이번 120 trial 은 느린 보행 기준).

### 참여자·세션 식별 (일반화 성능 왜곡 방지, 리뷰 #166)

같은 사람의 반복 trial 은 배치·보행 습관이 강하게 닮아 있다. 이를 무작위(행/파일 단위)로 train/eval 에
섞으면 **데이터 누수**로 시니어 일반화 성능이 과대평가된다. 이를 막으려면 **수집 시점에 참여자·세션을
식별**해 둬야 나중에 참여자 단위로 분할할 수 있다(사후 복원 불가).

- **수집 로그(별도 시트) 필수** — 매 trial 마다 아래를 기록한다. 키는 파일명(`trial_id`)이며 유일하다.

  | trial_id (파일명) | participant_id | session_id | 비고 |
  |---|---|---|---|
  | `walk_then_sit_20260723_...` | `P01` | `P01_s1` | (예) |

- **participant_id**: 익명 코드(`P01`, `P02`…) — **개인정보(이름·생년 등) 금지**.
- **session_id**: 같은 참여자라도 다른 날/다른 착복 상태면 세션을 나눈다(예: `P01_s1`, `P01_s2`).
- 최소 **참여자 수를 여러 명** 확보한다 — 1~2명이면 참여자 단위 분할 자체가 무의미하다.

> 이번 CSV 스키마에는 `participant_id` 컬럼이 없다(부록 A). 위 별도 로그로 관리하며, 자기기술(self-describing)
> 파일이 필요하면 후속에서 `participant_id`/`session_id` 를 메타 컬럼으로 승격한다(간단한 확장).

## 4. 폐기 규칙 (오염 데이터 배제)

아래 파일은 분석에서 **폐기**한다(수집 즉시 삭제 권장).

- `cue_delivery != success` 인 앉기 큐 파일 — 비프가 실제로 안 들렸을 수 있어 앉기 경계를 신뢰할 수 없다.
- `■ 중단(정지)`으로 끝낸 파일 — 화면 조작이 수집 구간에 섞였다.
- 백그라운드 진입으로 자동 정지된 파일 — 공백이 섞였다.
- 샘플 상한(30k) 자동 정지 파일 중 방치로 생긴 것.

> 큐 직후 짧은 정착 구간(비프·자세전환 ±300 ms)은 `excluded=1` 로 표시되므로, **폐기가 아니라
> 학습에서 제외**하면 된다(파일은 유지).

## 5. 분석 핸드오프 & 판별기 수용 기준 (후속 PR)

> **CSV 해석 주의(리뷰 #166):** `phase` 는 반드시 **(label, phase) 쌍**으로 읽는다 — `sit_only` 의
> `phase=walking` 은 '보행'이 아니라 **앉기 전 서있기 기준선**이다. '보행 구간'만 뽑으려면 phase 로만
> 거르지 말고 `label in (normal_walk, walk_then_sit)` 처럼 **라벨로 먼저** 걸러야 한다.

수집이 끝나면 라벨별 원시 파형에서 아래 특징을 탐색한다(리포 밖 CSV → 분석 노트북):

- **축 방향:** 앉기는 수직(중력축) 성분이 지배적, 정상 보행은 전후/좌우 성분이 더 큼 — 가설.
- **진폭·분산:** 앉기 하강 임팩트 vs 보행 스텝의 크기·산포 차이.
- **감쇠 패턴:** 보행 종료 후 신호가 잦아드는 곡선(별도 '보행 종료 감쇠' 상태 후보).

### 5-1. 1차 특징 스크리닝 스크립트 (`scripts/analyze_waveform.py`)

위 가설을 **수집 직후 바로** 정량 확인하는 표준 라이브러리 전용 스크립트(설치 불필요). 노트북에서
깊게 파기 전에 "어떤 특징이 실제로 두 동작을 가르는가"를 먼저 좁힌다.

```bash
# 파이프라인 자체검증(합성 데이터 — 데이터 없이 스크립트 동작 확인)
python scripts/analyze_waveform.py --selftest

# 실제 수집 CSV 디렉토리 분석
python scripts/analyze_waveform.py <CSV들이 든 디렉토리>

# 참여자 단위 교차표까지(권장) — trial_id,participant_id[,session_id] 로그를 함께
python scripts/analyze_waveform.py <디렉토리> --participants docs/waveform_participants.csv

# 수집 직후 파일 1개가 제대로 담겼는지 즉시 점검(현장 검증 — 현장 가이드 §6-1)
python scripts/analyze_waveform.py --inspect <파일.csv>
```

스크립트가 자동으로 하는 것(§4·§5 규칙 내장):

- **폐기 규칙**: 앉기 큐 라벨인데 `cue_delivery != success` 인 파일은 로드 단계에서 폐기, `excluded=1`
  행은 특징·오탐 집계에서 제외.
- **(label, phase) 세그먼트화**: `sit_only` 의 `phase=walking`(서있기 기준선)을 '보행'으로 오독하지
  않도록 **라벨로 먼저** 나눈다.
- **특징 계산(trial 단위 집계)**: `vert_share`(중력정렬 수직에너지 점유율 — **placement 불변**),
  `dyn_rms`·`fmag_std`(진폭·산포), `decay`(후/전반 에너지비 = 보행 종료 감쇠).
- **오탐 현황**: `walk_then_sit` 앉기 구간에서 잘못 세어진 걸음 수(= §5-5 의 +8 재현, 목표 ≤+2).
- **판별 스코어카드**: 음성 `normal_walk/walking` vs 양성 `walk_then_sit/sitting` 을 각 특징이 얼마나
  가르는지 **Cohen's d·추천 단일임계·오분류 수**로 출력, 가장 유망한 특징을 지목한다.
- **참여자 교차표**: `--participants` 로그가 있으면 라벨×참여자 분포와 leave-one-participant-out 가능
  여부를 찍는다.

> 이 스크립트는 **특징을 좁히고 임계 후보를 잡는 1차 도구**다. 단일 임계로 목표(≤+2)에 못 미치면
> 특징 조합·추가 센서(자이로/중력벡터)·별도 '보행 종료 감쇠' 상태를 노트북에서 이어 판다.

> **참여자 단위 분할(리뷰 #166):** 특징 탐색·임계 튜닝에서 train/eval 을 나눌 때는 **참여자 단위로**
> 분리한다(같은 `participant_id` 가 양쪽에 걸치지 않게). 무작위 행·파일 분할은 같은 사람의 닮은 trial 이
> 새어 들어가 실제 시니어 일반화 성능을 과대평가한다. 참여자가 적으면 leave-one-participant-out 로 본다.

판별기/상태머신 변경 PR 의 수용 기준(#89 회귀 없이):

| 시나리오 | 현재 | 목표 |
|---|---|---|
| 정상 보행 직후 앉기 5회 과다카운트 | +8 | **≤ +2** |
| 정상 보행 20보 | 18~22 | 유지 |
| 정지 상태 앉기 FP | 0 | **0 유지** |

> 이 문서·라벨 확장 PR 은 **`Refs #131`**(수집 도구·프로토콜까지). 판별기 튜닝과
> `WalkingStepDetectorAccuracyTest` 의 '알려진 한계' 테스트 갱신을 담은 후속 PR 이 #131 을 닫는다.

## 부록 A. CSV 컬럼 (`WaveformCsv.HEADER`)

```
trial_id,device_model,placement_id,position,side,screen_facing,top_direction,fold_state,
cue_delivery,label,phase,event,excluded,sensor_elapsed_ms,callback_elapsed_ms,
x,y,z,magnitude,filtered_mag,state,count,step_counted
```

- 시간축은 `sensor_elapsed_ms`(SensorEvent.timestamp 기준, 첫 샘플 = 0)를 기본으로, `callback_elapsed_ms`
  (콜백 도착)와의 차이로 스케줄링 지터를 진단한다.
- `phase` 는 '큐 전 활동 / 큐 후 착석' 의미다(`sit_only` 의 walking 은 서있기 기준선 = 걷는다는 뜻이 아님).
- 실수는 `Locale.US` 고정(소수점 `.`)이라 로케일과 무관하게 열이 안 깨진다.
